KALSHI INTEGRATION (500 lines total):

  kalshi-worker.js (500 lines)
    Role: Standalone HTTP server on port 3050
    Responsibility:
      • Load credentials from KALSHI-API-KEY.txt
      • Initialize REST client with kalshi-typescript SDK
      • Serve 11 endpoints (health, balance, markets, orders, etc.)
      • Cache market data locally
      • Handle errors gracefully
    Key feature: Detached from Electron (doesn't block UI)

  kalshi-ipc-bridge.js (170 lines)
    Role: Bridge Electron main ↔ worker subprocess
    Responsibility:
      • Spawn worker process on app startup
      • Wait for worker to be ready (health check)
      • Register 10 IPC message handlers
      • Proxy HTTP requests to worker
      • Kill worker on app shutdown
    Key feature: Enables renderer to call Kalshi API via IPC

  kalshi-renderer-bridge.js (60 lines)
    Role: Global window.Kalshi API for React/Vue
    Responsibility:
      • Expose methods: getBalance(), getMarkets(), placeOrder(), etc.
      • Wrap ipcRenderer calls
      • Cache market data in window._kalshiSnapshot
    Key feature: Simple, lightweight interface to Kalshi

  kalshi-rest.js (520 lines)
    Role: Wrap kalshi-typescript SDK
    Responsibility:
      • Initialize with RSA-2048 credentials
      • Implement all Kalshi API endpoints
      • Handle errors and map to user-friendly messages
      • Track statistics (calls, errors)
    Key feature: Hides SDK complexity from worker

                    KEY CONCEPTS TO UNDERSTAND
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 7-STATE SPIN MODEL
   Instead of binary UP/DOWN, your system uses:
   -3 (Strong Bear) → -2 (Bear) → -1 (Weak Bear) → 0 (Neutral) →
   +1 (Weak Bull) → +2 (Bull) → +3 (Strong Bull)
   
   Each state has base confidence (0.60-0.95) that gets adjusted.

2. KALSHI BLENDING
   Kalshi market odds (0-100) are converted to spin states.
   When CFM (your model) + Kalshi agree:
     • Confidence +25%
     • Order size +30%
   When they disagree:
     • Confidence -30%
     • Order size -50%

3. VOLATILITY REGIMES
   Market behavior changes based on ATR:
   • Tight (choppy): Require 40% higher entry signal
   • Normal: Standard behavior
   • Elevated: Slightly more aggressive
   • Extreme: Very conservative
   
   Each regime changes confidence caps and order sizing.

4. CONFIDENCE STACKING
   Final confidence = base × kalshiBoost × regimeScore × weakSignalPenalty (capped)
   
   Multiple independent adjustments compound together.

5. ORDER EXECUTION SIZING
   Order size = baseSize × spinMultiplier × blendingMultiplier × regimeMultiplier
   
   No fixed position size. Dynamic based on signal strength + market regime.

6. CONSENSUS SCORING
   When 4 models (CFM, Kalshi, derivs, structure) agree:
   • 4/4 models: 1.8x confidence boost
   • 3/4 models: 1.5x confidence boost
   • 2/4 models: 1.25x confidence boost
   
   Multiple independent signals = higher confidence.

7. CHOPPY MARKET FILTERS
   In tight regimes, weak signals get flattened:
   • |spinState| < 1.5 → signal = 0 (skip)
   • confidence -= 20%
   
   Prevents false trades in range-bound markets.

8. MULTI-PROCESS ISOLATION
   Credentials only in worker (not Electron, not renderer).
   UI won't block on API calls (separate process).
   Can restart worker without restarting app.
